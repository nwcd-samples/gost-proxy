# Kiro 网络环境检查工具 — 检测逻辑说明

用于在使用 Kiro 前，检测本机网络出口与代理配置是否满足 Kiro 的合规要求，避免因出口 IP 位于中国大陆/港澳而导致 **登录被封号** 或 **服务无法访问**。

| 文件 | 平台 | 运行方式 |
|---|---|---|
| `CheckKiroNetwork.sh` | macOS / Linux | `bash CheckKiroNetwork.sh` |
| `CheckKiroNetwork.ps1` | Windows | `powershell -ExecutionPolicy Bypass -File CheckKiroNetwork.ps1` |

两个版本检测逻辑完全一致。

---

## 一、核心概念

### 1. 合规区域 vs 风险区域

脚本判断一个 IP 是否"合规"，依据其所在国家/地区代码：

| 类型 | 区域代码 | 说明 |
|---|---|---|
| **风险区域** | `CN`、`HK`、`MO` | 中国大陆、香港、澳门 — 出口位于此处存在封号风险 |
| **合规区域** | 其他所有(US、JP、SG、DE…) | 海外出口 — 无风险 |

> 风险区域列表在脚本顶部 `RISK_REGIONS` / `$RiskRegions` 中定义，可自行增减。

### 2. 出口 IP 的获取

优先使用 **AWS 官方 `checkip.amazonaws.com`**（国内可直连、返回纯 IP），备用 `ipinfo.io`、`api.ip.sb`、`myip.ipip.net`。

> 之所以不用 `ifconfig.me` / `ip-api.com` 做主力：这两个服务在中国大陆直连常被拦截，会导致"获取不到出口 IP"的误判。

### 3. IP 区域识别

优先使用**地理数据库**类服务（反映 IP 的**实际服务位置**），`whois` 仅作最后兜底：

1. `ipinfo.io/{ip}/country`（地理库，国内可直连，最准）
2. `api.ip.sb/geoip/{ip}`（地理库，需带 User-Agent，国内可直连）
3. `ip-api.com`（地理库，通过代理 / 直连）
4. `whois`（**注册地** RIR 信息，最后兜底）

> **为什么不把 whois 放第一位？** whois 返回的是 IP 段的**注册国家**（分配给哪个机构），不是它**实际的服务位置**。例如某些云厂商的海外节点，IP 段注册在 A 国、实际部署在 B 国；`1.1.1.1` 的 whois 是澳大利亚（注册地），但它是全球 anycast。基于注册地判断会误报。因此改用地理数据库优先，`whois` 仅在所有地理库都不可用时兜底。

## 二、三项检查

### 检查 1：网络代理配置

列出当前系统的代理配置来源，仅作信息展示：

- **系统代理**
  - macOS：`scutil --proxy`（HTTP / HTTPS / SOCKS / PAC）
  - Windows：注册表 `Internet Settings`（ProxyEnable / ProxyServer / AutoConfigURL）
- **环境变量代理**：`HTTP_PROXY` / `HTTPS_PROXY` / `http_proxy` / `https_proxy`

结论：检测到任一代理即为"已配置"，否则"未检测到代理配置"。

---

### 检查 2：互联网出口 IP 区域检测（用于 Kiro 登录）

**目的**：Kiro **登录** 走的是系统默认浏览器。此项检测浏览器登录时的出口是否合规。

**流程**：

1. 识别默认浏览器（macOS LaunchServices / Windows 注册表 UserChoice）
2. 检测系统网络接口代理（浏览器通常跟随此设置）
3. 按浏览器实际网络路径依次尝试获取出口 IP：
   1. 系统网络接口代理
   2. 直连
   3. 环境变量代理
   4. 代理服务器地址（白名单代理兜底）
4. 识别出口 IP 区域并判断合规性

**结论**：

- 出口位于合规区域 → 无风险
- 出口位于风险区域(CN/HK/MO) → **存在封号风险**

#### ⚠️ 重要局限：浏览器扩展代理不可探测

如果浏览器通过 **扩展代理**（如 SwitchyOmega）上网，该代理运行在浏览器内部，**操作系统和脚本都无法探测**。此时脚本测到的是"系统直连出口"，可能与浏览器真实出口不同。

> **自检方法**：直接在浏览器中打开 `https://checkip.amazonaws.com`，显示的 IP 才是浏览器登录 Kiro 时真正使用的出口。

curl（脚本）与浏览器的差异，本质不在于"HTTP 请求本身"，而在于两者**读取代理配置的方式不同**：

| 代理方式 | curl 是否遵循 | 浏览器是否遵循 | 脚本能否探测 |
|---|---|---|---|
| 系统代理 | 否（脚本手动传给 curl 模拟） | 是 | ✅ |
| 环境变量代理 | 是 | 通常否 | ✅ |
| 浏览器扩展代理 | 否 | 是 | ❌ 无法探测 |
| PAC 自动代理 | 否 | 是 | ⚠️ 仅能读到地址，无法执行决策 |
| 直连 | 是 | 是 | ✅ |

---

### 检查 3：Kiro IDE 网络配置

**目的**：Kiro IDE（基于 Electron）访问后端服务时的出口是否合规。

#### 生效路径优先级（关键）

Kiro IDE 的**有效代理**按以下优先级确定，**只评估实际生效的那一条**：

```
Kiro IDE http.proxy  >  系统代理(回退)  >  直连
```

- IDE 配置了 `http.proxy` → 使用它（忽略系统代理）
- IDE 未配置 → 回退使用系统代理
- 两者都无 → 直连

#### 风险判定矩阵

| # | IDE http.proxy | 系统代理 | 生效路径 | 结论 |
|---|---|---|---|---|
| 1 | 已配置 · 合规 | 任意 | IDE proxy | ✅ 合规 — 无风险 |
| 2 | 已配置 · **不合规** | 合规 | IDE proxy | ❌ **有封号风险**（IDE 优先级最高，合规的系统代理不会被使用） |
| 3 | 已配置 · 合规 | 不合规 / 未配置 | IDE proxy | ✅ 合规 — 无风险（系统代理状态不影响） |
| 4 | 未配置 | 合规 | 系统代理 | ✅ 合规 — 无风险 |
| 5 | 未配置 | 不合规 | 系统代理 | ❌ 有封号风险 |
| 6 | 未配置 | 未配置 · 直连出口合规 | 直连 | ✅ 合规 — 无风险 |
| 7 | 未配置 | 未配置 · 直连出口不合规 | 直连 | ❌ 有封号风险 |

> **核心原则**：`IDE proxy` 优先级最高。场景 2 是高频误区 —— 即使系统代理是合规的海外节点，只要 IDE 自己配了一个不合规的 proxy，Kiro IDE 就会走这个不合规的代理，依然有封号风险。

#### 附加检测

- **连通性测试**：通过生效代理连接 `https://app.kiro.dev`，验证代理实际可用。
- **配置文件路径**：
  - macOS：`~/Library/Application Support/Kiro/User/settings.json`
  - Windows：`%APPDATA%\Kiro\User\settings.json`

## 三、总结：Kiro 使用风险结论

三项检查跑完后，脚本会输出一个**综合结论**，分别针对两个独立场景给出判定：

| 场景 | 依据 | 判定来源 |
|---|---|---|
| **Kiro 登录** | 默认浏览器出口区域 | 检查 2 |
| **Kiro IDE** | 生效代理出口区域 | 检查 3 |

每项判定为三态之一：

- `✓ 无风险`：出口合规（海外）
- `✗ 有封号风险`：出口位于 CN/HK/MO
- `⚠ 待确认`：无法探测（如浏览器扩展代理），需手动核实

**总体结论**：

- 任一项为"有封号风险" → **总体: 存在封号风险，请先修正后再使用 Kiro**
- 两项均"无风险" → **总体: 均合规，可安全使用 Kiro**
- 否则 → **总体: 存在待确认项，请手动核实**

> Kiro 登录（浏览器）和 Kiro IDE（Electron 代理）走的是**两条独立的网络路径**，必须**分别满足合规**。任何一条不合规都可能导致封号。

## 四、安全提示

- 脚本对代理 URL 中的密码做了**脱敏显示**（`user:****@host`），不会明文打印密码。
- 但请注意：如果你的代理凭证以明文存放在 Kiro `settings.json` 或环境变量中，本身即存在泄露风险，建议定期轮换。

## 五、常见问题

**Q：检查 2 报"无法获取出口 IP"，但浏览器能正常上网？**
A：很可能你的浏览器使用了扩展代理（SwitchyOmega 等），脚本探测不到。请在浏览器中打开 `https://checkip.amazonaws.com` 自行确认出口。

**Q：检查 2 显示 CN 有风险，但检查 3 显示无风险？**
A：正常。检查 2 看的是"浏览器出口"（登录用），检查 3 看的是"Kiro IDE 生效代理出口"。两者走的网络路径可能不同，需分别满足合规。

**Q：为什么用 `checkip.amazonaws.com`？**
A：它是 AWS 官方 IP 回显服务，返回纯 IP、稳定，且在中国大陆可直连；而 `ifconfig.me` / `ip-api.com` 在国内常被拦截。
